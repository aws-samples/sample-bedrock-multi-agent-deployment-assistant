# Configuration Guide

This document describes the two configuration files that drive the AI Deploy Assistant, all their types, and how they interact with the runtime system.

## Overview

```
config.yaml (hand-edited, ~10 lines)
    ↓
catalog.lock.yaml (generated from KB, committed to repo)
    ↓
CatalogLoader (singleton, provides typed access at runtime)
    ↓
All agents, services, and prompts consume catalog context
```

---

## config.yaml — Product Identity

The minimal configuration file. This is the only file you edit by hand.

```yaml
product:
  name: "NovaMind Inference Server"   # Display name used in prompts and UI
  vendor: "NovaMind AI"               # Vendor name for KB queries

knowledge_base:
  id: "ABC123XYZ"                     # Bedrock KB ID (production)
  local_path: "knowledge-base"        # Local docs folder (development fallback)

overrides:
  validation:
    cfn_guard_rules_file: "path/to/rules.guard"  # Custom cfn-guard rules
    checkov_skip:
      - "CKV_AWS_23"                  # Check IDs to skip
    checkov_skip_rationale:
      CKV_AWS_23: "Reason for skipping"
```

### Schema Types

#### `ProductConfig`
| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `name` | `str` | Yes | Product display name — injected into prompts as `{product_name}` |
| `vendor` | `str` | Yes | Vendor name — injected into prompts as `{vendor_name}` |

#### `KnowledgeBaseConfig`
| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | `str \| null` | No | AWS Bedrock Knowledge Base ID. When set, uses Bedrock for search. |
| `local_path` | `str \| null` | No | Path to local KB directory (relative to project root). Used when `id` is empty. |

#### `ValidationOverrides`
| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `cfn_guard_rules_file` | `str \| null` | `null` | Path to custom cfn-guard rules file |
| `checkov_skip` | `list[str]` | `[]` | Checkov check IDs to skip during validation |
| `checkov_skip_rationale` | `dict[str, str]` | `{}` | Human-readable reasons for each skip |

## catalog.lock.yaml — Product Schema

The deterministic contract for the deployment assistant. Generated by the Catalog Discovery Agent from the Knowledge Base, then reviewed and committed. At runtime, this file determines:

- What products exist in the catalog
- What interview fields to collect (blocking vs optional)
- What deployment patterns are available
- Appliance configuration (instance types, interfaces)
- KB search templates and guardrails

### Top-Level Structure

```yaml
schema_version: 1

products:
  - id: novamind-inference        # Machine ID
    name: NovaMind Inference Server
    vendor: NovaMind AI
    category: ml-infrastructure
    kb_doc_prefix: ""             # S3/local path prefix for product docs

    interview: { ... }           # InterviewSchema
    appliance: { ... }           # ApplianceConfig
    patterns: [ ... ]            # list[PatternConfig]

knowledge_base:
  search_template: "{product_name} {use_case} deployment AWS"
  document_types: [architecture, sizing, configuration, components, best-practices]

guardrails:
  focus_topics: ["ML inference", "AWS services"]
  denied_topics: ["non-catalog products"]
```

### Schema Types (Full Reference)

---

#### `CatalogLock` (root)
| Field | Type | Description |
|-------|------|-------------|
| `schema_version` | `int` | Schema version for forward compatibility (currently `1`) |
| `products` | `list[CatalogProduct]` | All products in the catalog |
| `knowledge_base` | `KBSearchConfig` | KB search configuration |
| `guardrails` | `GuardrailsConfig` | Content guardrail settings |

---

#### `CatalogProduct`
| Field | Type | Description |
|-------|------|-------------|
| `id` | `str` | Unique machine identifier (e.g., `novamind-inference`) |
| `name` | `str` | Human-readable product name |
| `vendor` | `str` | Vendor/manufacturer name |
| `category` | `str` | Product category (e.g., `ml-infrastructure`, `network-security`) |
| `kb_doc_prefix` | `str` | Path prefix for this product's KB documents |
| `interview` | `InterviewSchema` | Complete interview field definitions |
| `appliance` | `ApplianceConfig` | Hardware/instance configuration |
| `patterns` | `list[PatternConfig]` | Available deployment patterns |

---

#### `InterviewSchema`
| Field | Type | Description |
|-------|------|-------------|
| `use_cases` | `list[UseCaseSchema]` | Product use cases with their fields |
| `base_fields` | `BaseFieldsSchema` | Fields that apply regardless of use case |

---

#### `BaseFieldsSchema`
| Field | Type | Description |
|-------|------|-------------|
| `blocking` | `list[FieldDefinition]` | Fields that MUST be answered before design can proceed |
| `soft` | `list[FieldDefinition]` | Fields the interview tries to collect, but user decline is OK |

---

#### `UseCaseSchema`
| Field | Type | Description |
|-------|------|-------------|
| `value` | `str` | Machine key (e.g., `realtime-inference`) — used in API payloads |
| `label` | `str` | Human-readable label (e.g., `Real-Time Inference`) — shown in UI |
| `required_fields` | `list[FieldDefinition]` | Fields that block completion for this use case |
| `optional_fields` | `list[FieldDefinition]` | Fields to collect but accept a decline |

---

#### `FieldDefinition`

The core building block for interview fields. Each field has a type, optional constraints, and a KB document type for targeted search.

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `name` | `str` | — | Field identifier (e.g., `model_size_category`) |
| `type` | `str` | — | Type hint (see table below) |
| `description` | `str` | `""` | Human-readable description (shown to LLM for question phrasing) |
| `options` | `list[str] \| null` | `null` | Valid values for `enum` types |
| `unit` | `str \| null` | `null` | Unit of measurement (e.g., `Mbps`, `GB`) |
| `default` | `any` | `null` | Default value if user doesn't specify |
| `doc_type` | `str` | `"configuration"` | KB document_type for targeted search |
| `fields` | `list[FieldDefinition] \| null` | `null` | Sub-fields for `object` types |

**Supported `type` values:**

| Type | Python Equivalent | Example |
|------|-------------------|---------|
| `str` | `str` | Free-text input |
| `int` | `int` | Whole numbers |
| `float` | `float` | Decimal numbers |
| `bool` | `bool` | True/false |
| `enum` | `str` (validated against `options`) | Single choice from list |
| `list_str` | `list[str]` | Multiple string values |
| `object` | `dict` | Nested structure (uses `fields` for sub-fields) |

---

#### `ApplianceConfig`
| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `interface_naming` | `str` | `"port{n}"` | Interface naming pattern (template variable `{n}`) |
| `approved_instance_types` | `list[str]` | `[]` | EC2 instance types allowed for this product |
| `interface_roles` | `dict[str, InterfaceRoleConfig]` | `{}` | Named network interface roles |

---

#### `InterfaceRoleConfig`
| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `source_dest_check` | `bool` | `true` | AWS SourceDestCheck setting for this interface role |
| `subnet_role` | `str` | — | Which subnet role this interface connects to |
| `description` | `str` | `""` | Human-readable purpose of this interface |

---

#### `PatternConfig`
| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `name` | `str` | — | Primary pattern name (e.g., `auto-scaling-fleet`) |
| `aliases` | `list[str]` | `[]` | Alternative names for fuzzy matching |
| `layers` | `list[str]` | `[]` | Layer names this pattern uses (e.g., `[foundation, security, compute, scaling]`) |
| `description` | `str` | `""` | Human-readable pattern description |

---

#### `KBSearchConfig`
| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `search_template` | `str` | `"{product_name} {use_case} deployment AWS"` | Template for KB queries (variables substituted at runtime) |
| `document_types` | `list[str]` | `[architecture, sizing, ...]` | Valid document type categories |

---

#### `GuardrailsConfig`
| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `focus_topics` | `list[str]` | `[]` | Topics the assistant should focus on |
| `denied_topics` | `list[str]` | `[]` | Topics the assistant should refuse |

## Runtime: CatalogLoader

The `CatalogLoader` is a singleton initialized at startup from `catalog.lock.yaml`. It provides typed access methods:

```python
from src.services.catalog_loader import get_catalog

catalog = get_catalog()

# Product access
catalog.default_product              # CatalogProduct
catalog.get_product("novamind-inference")

# Use case access
catalog.get_use_cases()              # list[UseCaseSpec]
catalog.get_use_case_spec("realtime-inference")
catalog.get_use_case_values()        # ["realtime-inference", "batch-inference", "training"]

# Field access
catalog.get_blocking_base_fields()   # list[FieldDefinition]
catalog.get_soft_base_fields()       # list[FieldDefinition]
catalog.get_fields_for_use_case("realtime-inference")  # (required, optional)

# Pattern access
catalog.get_patterns()               # list[PatternConfig]
catalog.resolve_pattern("fleet")     # PatternConfig (fuzzy match by name/alias)

# Appliance config
catalog.get_approved_instance_types()  # ["g5.xlarge", "g5.2xlarge", ...]
catalog.get_interface_roles()        # dict of role configs

# Prompt context (injected into all prompt templates)
catalog.get_prompt_context()         # {"product_name": ..., "vendor_name": ..., ...}

# KB search
catalog.format_search_query("realtime-inference")  # "NovaMind Inference Server realtime-inference deployment AWS"
```

## Runtime: KB Provider

The `KnowledgeBaseProvider` is selected automatically based on configuration:

```python
from src.services.kb_provider import get_kb_provider

provider = get_kb_provider()
results = provider.search("GPU inference auto scaling", max_results=5, use_case="realtime-inference")
```

### Provider Selection Priority

1. `AI_DEPLOY_KNOWLEDGE_BASE_ID` env var set → **BedrockKBProvider**
2. `knowledge_base.local_path` in config.yaml → **LocalKBProvider**
3. Neither → **NullKBProvider** (returns empty results)

### Local KB Directory Structure

```
knowledge-base/
  {use_case}/              # Matches use case values from catalog
    {deployment_type}/     # Sub-category (e.g., "single-gpu", "auto-scaling")
      {document_type}.md   # Matches document_types from KB config
```

Example:
```
knowledge-base/
  realtime-inference/
    single-endpoint/
      architecture.md
      sizing.md
      configuration.md
    auto-scaling/
      architecture.md
  batch-inference/
    single-gpu/
      architecture.md
      sizing.md
      configuration.md
  training/
    distributed/
      architecture.md
    single-node/
      architecture.md
```

The local provider performs TF-IDF text search with path-based metadata filtering — same semantics as Bedrock's vector search with metadata filters.

## Prompt Template Variables

All prompt files in `backend/src/prompts/*.txt` support these variables (injected from catalog):

| Variable | Source | Example Value |
|----------|--------|---------------|
| `{product_name}` | `config.yaml → product.name` | `NovaMind Inference Server` |
| `{vendor_name}` | `config.yaml → product.vendor` | `NovaMind AI` |
| `{interface_naming_pattern}` | `catalog → appliance.interface_naming` | `eni-{role}` |
| `{approved_instance_types}` | `catalog → appliance.approved_instance_types` | `g5.xlarge, g5.2xlarge, ...` |

Runtime-specific variables (filled per request):
| Variable | Filled By | Example |
|----------|-----------|---------|
| `{seed_context_block}` | Interview planner | Use cases + bandwidth from wizard |
| `{kb_results}` | KB search results | Formatted search results |
| `{missing_fields_schema}` | Catalog field definitions | JSON schema of uncollected fields |
| `{layer_name}` | IaC agent per-layer | `compute` |
| `{allowed_resource_types}` | IaC agent per-layer | `AWS::EC2::Instance, AWS::EC2::LaunchTemplate` |

## Adding a New Product

To support a new product with this system:

1. **Create `config.yaml`** with product name, vendor, and KB connection
2. **Populate Knowledge Base** — upload product docs to Bedrock KB (or place in `knowledge-base/` locally)
3. **Create `catalog.lock.yaml`** — define use cases, fields, patterns, and appliance config (use the existing file as a template)
4. **Review and commit** the lock file
5. **Optionally**: add predefined LayerPlan patterns in `backend/src/services/predefined_layers.py` for common deployments (saves ~5-8s per IaC generation by skipping the LLM Architecture Planner)
6. **Optionally**: add custom cfn-guard rules in `backend/src/validation/appliance_rules.guard`

The entire interview → design → IaC → docs pipeline will automatically adapt to the new product's schema.
