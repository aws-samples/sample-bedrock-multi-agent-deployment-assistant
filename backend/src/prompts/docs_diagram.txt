Generate a Mermaid architecture-beta diagram for the following CloudFormation template.

This is a network and security architecture diagram for a FortiGate-VM deployment
on AWS. It must be clean, organized, and professional-looking.

=== OUTPUT ===
Output ONLY the raw Mermaid code. No markdown fences, no explanation, no preamble.

=== LAYOUT STRATEGY ===
The diagram MUST follow a strict LEFT-TO-RIGHT network flow:
1. LEFTMOST: Internet-facing resources (Internet Gateway, ALB, CloudFront)
2. CENTER-LEFT: Public subnets with FortiGate instances and NAT Gateways
3. CENTER-RIGHT: Private subnets with application workloads
4. RIGHTMOST: Data layer (RDS, DynamoDB, S3) and IAM/monitoring resources

Organizing rules:
- Group resources by FUNCTION, not just by AWS type
- Each VPC is a top-level group; subnets are nested inside
- Keep the diagram COMPACT — avoid unnecessary whitespace
- Place related resources close together within the same group
- Resources not bound to a VPC (IAM Roles, S3 Buckets) go outside all groups
- Maximum 15-20 service nodes — combine similar resources
  (e.g. show "FortiGate HA Pair" instead of two separate FortiGate nodes)

=== EDGE RULES ===
Edges control the visual layout. Follow these rules for clean diagrams:
- Use ONLY short single-hop edges: source:R -- L:target (left-to-right flow)
- For top-to-bottom connections: source:B -- T:target
- NEVER chain long paths — only connect directly related resources
- Limit to 8-12 edges total — show the PRIMARY data flow, not every reference
- Edge priority (include these first):
  1. Internet -> FortiGate (ingress path)
  2. FortiGate -> backend services (inspected traffic)
  3. Backend -> data stores (application flow)
  4. FortiGate -> IAM role (control plane)
- Skip edges for: security group associations, log destinations, parameter references

=== CRITICAL SYNTAX RULES ===
1. First line MUST be exactly: architecture-beta
2. ALL element IDs MUST start with "r_" prefix (e.g. r_MyVpc, r_FortiGate)
3. IDs may only contain [a-zA-Z0-9_] — no hyphens, dots, or spaces
4. Groups: group <id>(<icon>)[<label>]  or  group <id>(<icon>)[<label>] in <parent_id>
5. Services: service <id>(<icon>)[<label>]  or  service <id>(<icon>)[<label>] in <group_id>
6. Edges: <service_id>:<direction> -- <direction>:<service_id>
   Directions: L (left), R (right), T (top), B (bottom)
7. Edges can ONLY connect services — NEVER connect to groups
8. Use " -- " for edges (two hyphens with spaces). NEVER use "-->" arrows.
9. Every service/group ID must be unique
10. LABELS: Text inside square brackets [<label>] may ONLY contain letters,
    numbers, spaces, and hyphens. NEVER use dots, slashes, colons, underscores,
    brackets, parentheses, or any special characters in labels.
    Do NOT include IP addresses, CIDR blocks, or port numbers in labels.
    WRONG: [Public Subnet 10.0.0.0/24]  CORRECT: [Public Subnet AZ1]
    WRONG: [FortiGate ENI (port1)]  CORRECT: [FortiGate ENI Port1]
